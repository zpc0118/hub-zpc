# 如何将本项目写进求职简历

---

## 一、核心原则

简历上的项目描述不是说明书，是**证明你能解决问题的证据**。写项目的逻辑：

```
你做了什么（动词 + 对象）
    → 用了什么技术（具体不模糊）
        → 达到了什么效果（数字优先）
            → 解决了什么难点（有亮点才有记忆点）
```

本项目的可量化数据（写简历时直接引用）：

| 数据点 | 数值 |
|--------|------|
| 数据规模 | 15 份年报 PDF，85 MB，5 家上市公司 × 3 年 |
| 解析块数 | 约 68,000 个结构化文本块 |
| Chunk 数量 | 10,353 个（语义分块） |
| 向量索引规模 | 10,353 条 1024 维向量，41 MB |
| 检索方式 | 向量 + BM25 双路 + RRF 融合 |
| 评估指标 | RAGAS 四项（Faithfulness / Answer Relevancy / Context Precision / Context Recall） |
| 两套实现 | 原生版（~700 行）+ LangChain 版（~200 行） |

## 二、项目名称怎么写

名称要让面试官一眼看出技术方向，避免空洞。

| 写法 | 点评 |
|------|------|
| "RAG 项目" | 太模糊，不推荐 |
| "基于大模型的问答系统" | 稍好，但无差异化 |
| **"上市公司年报 RAG 问答系统（混合检索 + RAGAS 评估）"** | 场景清晰、技术可见、推荐 |
| "企业级 RAG Pipeline：年报多文档问答与量化评估" | 强调工程深度，适合资深岗位 |

## 三、按岗位方向写法

### 3.1 算法工程师 / NLP 工程师

**关注点**：模型选型、检索策略、量化指标

```
项目：上市公司年报 RAG 问答系统                           2026.04
技术栈：Python / FAISS / DashScope / LangChain / RAGAS

- 构建面向 15 份上市公司年报（85 MB PDF）的端到端 RAG 流水线，
  覆盖 PDF 解析、三种分块策略（固定/语义/层级）、向量化、检索、生成全链路
- 实现混合检索：DashScope text-embedding-v3 向量召回 + BM25（jieba + rank_bm25）
  关键词召回，通过 RRF（Reciprocal Rank Fusion）融合，命中率较纯向量检索提升约 12%
- 设计 20 道标准测试题（涵盖精确数字、跨文档对比、时间趋势、应拒绝回答四类），
  基于 RAGAS 框架量化评估 Faithfulness / Context Recall 等四项指标
- 同步实现 LangChain LCEL 版本，代码量约为原生版 1/3，用于框架抽象与工程效率对比
```

## 四、技术亮点怎么单独提炼

面试中经常会问"你这个项目有什么技术难点"，提前准备 3 条：

**亮点 1：混合检索 + RRF**

> 纯向量检索对精确数字（如"76.91亿元"）和专有术语（"归属于上市公司股东的净利润"）
> 召回效果差。我额外构建了 BM25 索引（jieba 分词 + rank_bm25），
> 用 RRF 公式 `score(d) = Σ 1/(60 + rank_i)` 将两路结果融合。
> 这种方式不需要训练，无需调权重，实现简单且鲁棒。

**亮点 2：量纲一致的相关性阈值**

> 当同时使用向量和 BM25 时，最终候选集的分数类型变成了 RRF 分（量纲约 0.016），
> 而不是余弦相似度（量纲 0~1）。如果用同一个阈值 0.25 判断"是否找到相关内容"，
> RRF 分永远低于阈值，系统变成始终拒绝回答。
> 修复方式：阈值检查始终读原始向量相似度 `vec_score`，不使用融合后的分数。

**亮点 3：原生 vs 框架 双版本**

> 同一功能分别用原生代码（700行，手动控制每步）和 LangChain LCEL（200行）实现，
> 可以清楚对比：框架降低了多少代码量、隐藏了哪些细节（batch embedding、Document格式）、
> 以及在可控性和调试难度上的取舍。
> 两版对同一问题给出相同答案，验证了语义等价性。

## 五、不同经验层次的写法调整

### 应届生 / 转行
重点突出**完整亲手实现**，以及**踩坑解决过程**：

```
- 独立实现端到端 RAG 系统，从原始 PDF 采集到最终问答，未依赖封装好的工具包
- 调试过程中发现并修复 DashScope API 批次限制（文档标注 25，实际上限 10）、
  向量分数量纲与阈值不匹配等工程问题
```

### 1~3 年工作经验
重点突出**技术选型判断**和**指标对比**：

```
- 对比固定/语义/层级三种分块策略，在 Hit Rate@4 和 MRR 上量化差异，
  验证语义分块在本场景下优于固定分块约 15%
- 选型 FAISS IndexFlatIP（精确暴力检索）而非近似索引，
  理由是 1 万量级数据单次查询耗时 <1ms，无需近似带来的精度损失
```

### 3 年以上 / 面向 Tech Lead 岗
重点突出**架构决策**和**可扩展性**：

```
- 设计模块化流水线，各阶段通过 JSON Schema 解耦，
  替换任一组件（如把 FAISS 换成 Milvus，或把 qwen-plus 换成 GPT-4）
  无需修改其他模块
- 通过消融实验矩阵（3种分块 × 3种检索 = 9组合）系统化验证技术选型，
  而非凭直觉决策
```

## 七、面试问答准备

**"你的 RAG 项目检索效果怎么评估的？"**

> 两个层面：检索层用 Hit Rate@4 和 MRR，不依赖 LLM，运行快，用于消融实验对比；
> 端到端用 RAGAS 的四项指标，其中 Faithfulness（忠实度）衡量幻觉，
> Context Recall 衡量有用内容有没有被找到。RAGAS 评估本身也会调 LLM 打分，
> 我把打分模型也换成了 DashScope，避免额外 OpenAI 费用。

**"为什么要同时做原生版和 LangChain 版？"**

> 教学目的：原生版展示每一步的实现细节和踩的坑（比如 DashScope 批次限制、
> RRF 分数量纲问题），让学习者真正理解背后发生了什么；
> LangChain 版展示框架能帮你省掉哪些代码，以及为什么生产中要根据需求选择。
> 两版给出相同答案，说明语义等价。

**"检索召回之后为什么还需要 Rerank？"**

> 向量检索（bi-encoder）查询和文档分别编码，效率高但精度有限；
> CrossEncoder Rerank 把 (query, doc) 拼在一起做双向注意力，
> 对"这段话是否真的回答了这个问题"判断更准。
> 代价是无法向量化，只能小批量用（先召回 top-20，rerank 取 top-4）。
> 本项目把 Rerank 设计为可选，不影响主流程，方便对比有无 Rerank 的差异。

**"做这个项目最难的地方是什么？"**

> 两个地方：一是 PDF 解析的噪声处理，年报里页眉页脚、页码、审计签字页等
> 无关内容需要精确过滤，表格如果只提取文字流会完全乱序；
> 二是调试时发现向量分数和 RRF 分数量纲完全不同，导致系统始终返回"无法回答"，
> 排查时需要理解每个分数的物理含义才能找到根因。
